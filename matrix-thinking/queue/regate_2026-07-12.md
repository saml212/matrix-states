# Queue re-gate, 2026-07-12 — §11.2 verdict-driven

**Trigger:** `matrix-thinking/NOVEL_ARCH_WATERFALL.md` §11.2 ("EARLY-LN
K-SCALING RUN + HARVEST") landed the K-scaling ladder's pooled verdict:
**TRAINABILITY-STILL-LIMITED** (worst-of K∈{15,16,24}). K=15 moved
DEAD→SCALES (4/4 converged, holds far-depth composition at h\*=117); the
wall re-forms at K=16 (1/4 converged, dirty writes) and K=24 (0/4,
partial-formation profile — loss 1.0→0.36, A_eff_rank→17.7/24). §11.2's
own closing paragraph: the queue's Lane-A K∈{20..256} ladder "AS CURRENTLY
SPECIFIED will — on this verdict's evidence — measure TRAINABILITY-DEAD
cells at every rung ≥24 under the 80K/flat-anneal recipe and should be
re-gated on (i)/(ii) first" (i)=K16 budget/anneal probe, (ii)=K24 same,
both priced (~3.5 / ~4 GPU-h) but not launched. K=20 has no readout yet —
genuinely open, kept.

## 1. Park (removals)

All **pending** (never claimed/completed/failed) Lane-A "main" K-ladder
cells with K≥24, and Lane-C "seed-deepening" cells for K≥24, moved
`~/queue/pending/` → new `~/queue/parked_k24plus/` on the box. Reversible
— `mv` back into `pending/` to re-activate once a budget/anneal fix (or
other architecture change) is validated at K=16/24 first.

- Lane A parked: 20 jobs (K=48×4, K=64×4, K=96×4, K=128×4, K=192×2,
  K=256×2) — 81.322 GPU-h estimated.
- Lane C parked: 14 jobs (K=32×2, K=48×2, K=64×2, K=96×2, K=128×2,
  K=192×2, K=256×2) — 62.708 GPU-h estimated.
- **Total parked: 34 jobs, 144.030 GPU-h.**

Note: 2 K=32 main cells (IDs `106`,`107`) were claimed by workers in the
few seconds between inventory and the `mv` (an ordinary queue race, not an
error) — per the standing rule, claimed jobs are never touched, so they
were correctly left running. K=32 was never itself part of the §11.2
K-ladder readout (K∈{14,15,16,24} only), so these 2 cells add a genuine
new (if now deprioritized) K=32 datapoint at no cost beyond what was
already committed.

Kept, unparked: all K=20 jobs (Lane A main, already claimed/completing;
Lane C deepen), all Lane B jobs (untouched — K-independent), all rate
probes (cheap, informative, several already completed).

## 2. Refill (additions)

`matrix-thinking/queue/generate_jobs.py` extended (additive only — no
existing job-generating function for IDs 000-315 modified; independently
re-verified byte-identical on re-generation, see Audit below) with three
new functions, called from `regate_20260712_jobs()`:

**(a) The two §11.2-priced probes** (`lane_a_budget2x_probe_jobs()`, IDs
`050`-`057`, front of pending, priority below the current lowest
remaining Lane-A pending prefix at the time):

- K=16, `--steps 160000` (2× `STEPS_MAIN=80_000`), seeds 0-3.
- K=24, `--steps 160000`, seeds 0-3.

`ncr_earlyln_scale.py`'s `ln_alpha_at(step, total)` anneals linearly over
`total // 2`, where `total` is exactly the `--steps` CLI value
(`ncr_task.py`/`ncr_earlyln_scale.py:147-152`, confirmed by independent
audit) — so a single `--steps 160000` call is simultaneously a 2× budget
**and** a 2× anneal-length probe. No new CLI knob needed; no
model/training code touched, satisfying the "do not modify model/training
logic" constraint.

Cost basis: **measured**, not formula-extrapolated — §11.2's own real
per-K 4-seed 80K-step totals (K16 1.705 GPU-h, K24 2.015 GPU-h) divided by
4 and linearly doubled: K16 0.8525 GPU-h/cell (3.41/4 seeds), K24 1.0075
GPU-h/cell (4.03/4 seeds) — matches §11.2's own disclosed "~3.5"/"~4"
GPU-h estimate. Ceiling set to 4× the 80K per-cell rate (≥99% margin over
the 2×-linear estimate).

**Separate outdir, load-bearing (not cosmetic):** these jobs write to a
new `~/ncr/results_earlyln_budget2x/`, NOT the main
`~/ncr/results_earlyln_scale/`. The script's own whole-cell
skip-if-COMPLETED resume check (`ncr_earlyln_scale.py:217-222`) returns
the pre-existing record for a `(K,seed)` pair **regardless of the
requested `--steps`** — reusing the main outdir would have silently
returned the already-completed 80K-step record unrun, and the
`validity_check`'s `train.step==160000` assertion would then have
correctly routed the job to `failed/`, burning the slot without measuring
anything. Verified empirically before deploy: `earlyln_K16_s{0..3}.json`
and `earlyln_K24_s{0..3}.json` already exist COMPLETED in the main outdir;
the new outdir was empty.

**(b) Depth refill** to restore pending+claimed toward the ≥384 GPU-h
(2-day / 8-GPU / 24h) target, using only already-audited machinery:

- `lane_b_seedext_increment_jobs()` (IDs `400`-`431`, 32 jobs): next-seed
  extension of the CLOSED S13.22-style seedext cells already in the queue
  — 392M seeds 8-11 (was 4-7) and 98M seeds 7-10 (was 3-6), both arms
  (`off`/`per_token`), both corpora, identical `_fixscale_cell()` call and
  cost basis (392M: `ref_per_step_s=0.836` → 4.671 GPU-h/cell; 98M:
  `ref_per_step_s=0.236` → 4.478 GPU-h/cell). 32 × ~4.5-4.7 ≈ 148.9 GPU-h.
- `lane_c_k20_increment_jobs()` (IDs `450`-`451`, 2 jobs): K=20 deepened
  n=6→n=8 (seeds 6,7) — K=20 is the one rung §11.2 leaves genuinely open.
  ≈0.826 GPU-h.

No probe was skipped for a missing CLI knob — both §11.2-priced probes
were fully buildable with existing flags.

Generated locally (`DRY_RUN_BYPASS=1 python3 generate_jobs.py`, the
project's pre-train-gate hook's documented bypass for a non-training
generator script that was mis-matched by the naive `*.py` pattern), then
the 42 new files only were `scp`'d into `~/queue/pending/` on the box —
the rest of `jobs/pending/` (unchanged, byte-identical) was **not**
re-deployed, avoiding any risk to already-claimed/completed/parked jobs.

## 4. Audit

Independent fresh-context audit (model: opus, read-only, no launches)
covered: park correctness, command runnability (CLI flags + K choices
verified against the real argparse tables on box), GPU-h estimate
justification (cross-checked against §11.2's raw per-cell `gpu_h` values
in the real result JSONs), validity-check correctness against real result
JSON schemas, timeout margins, ledger arithmetic, and repo/generator
diff cleanliness (byte-identical re-generation of pre-existing job IDs,
confirmed via md5 on 4 spot-checked files across all 3 lanes).

**Verdict: PASS on all 7 audit tasks. Zero FATAL, zero MAJOR.** Two MINOR
observations, both dispositioned, no code changes required:

1. The 2 late-claimed K=32 main cells (see §1) are a benign race, not a
   defect — correctly left running per the "never touch claimed" rule.
2. Lane-B `gpu_h_estimate` values (4.671 / 4.478) are inherited verbatim
   from the already-deployed originals and run ~0.6-1.1% above pure
   `steps × measured_rate / 3600` arithmetic — a safe, pre-existing
   conservative rounding, not a new issue.

## 5. What was NOT done

- No probe was parked for a missing CLI knob — none was missing.
- No model/training code was modified anywhere in this re-gate.
- `deploy.sh` (the full-redeploy script) was deliberately **not** run —
  its job-spec sync assumes a first-deploy / fully-static `pending/`, and
  running it against a live, partially-claimed/completed/parked queue
  would trip its own count-mismatch guard (a correct refusal, not a bug)
  or risk re-introducing already-consumed specs. The surgical
  scp-only-the-delta approach used here is the safe pattern for any
  future incremental refill against a live queue.
